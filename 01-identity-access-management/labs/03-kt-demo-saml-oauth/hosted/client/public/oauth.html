OAuth 2.0 + OIDC — the grant flows

← back  ·  each card is a real, interactive flow. Keep DevTools → Network open.

Which grant, in one breath: human in a browser → Authorization Code + PKCE (public) or no PKCE (confidential server app). No human (a service) → Client Credentials. No keyboard (TV/CLI) → Device Code. Implicit is the dead one (token in the URL). ROPC — the app takes your password — is deprecated publicly but still used for trusted first-party apps inside an org.

✅ the default · OIDC

1 · Authorization Code + PKCE →

Any app with a user. See the code_challenge, the one-time ?code=, the back-channel token swap, and the decoded access + ID tokens. Includes a Refresh button.

 confidential · no PKCE

1b · Authorization Code (no PKCE) →

The classic server-side web app flow. A confidential client proves itself with a client secret instead of PKCE; the code is exchanged on the server. Nice contrast to flow 1.

 ✅ machine-to-machine

2 · Client Credentials →

No user, no browser — the service authenticates as itself (run server-side). Get a token, then call a real Resource Server → 200. No refresh, no ID token.

 ✅ no keyboard

3 · Device Code →

Smart-TV / CLI flow: get a user_code, approve it on another tab, watch the device poll until a token drops.

 ☠ deprecated

4 · Implicit →

The old SPA way. The access token comes back in the URL fragment — see why that leaks, and why OAuth 2.1 removed it.

 legacy · trusted first-party

5 · ROPC (password) →

The app takes your username + password directly. Deprecated for public/third-party use, but still used inside orgs for trusted first-party apps (legacy tools, CLIs, migrations).
