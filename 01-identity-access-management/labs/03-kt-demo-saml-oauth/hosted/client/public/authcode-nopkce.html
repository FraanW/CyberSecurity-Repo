1b · Authorization Code — no PKCE (confidential client)

OAuth flows  ·  confidential client kt-web · code exchanged server-side with a secret

Confidential means the secret never leaves the server.

The classic server-side web-app flow (pre-PKCE default). A confidential client proves itself with a client secret instead of PKCE. Because the secret must never reach the browser, the code→token exchange happens on this app's server, not in the page.

Not logged in. Watch DevTools → Network: /authorize has no code_challenge this time; the token exchange happens on our server (/api/authcode-exchange), so the secret stays server-side.

PKCE vs no-PKCE: a public client (SPA/mobile) can't hide a secret, so it proves itself with PKCE (flow 1). A confidential server app can keep a secret, so historically it skipped PKCE — this flow. Modern guidance (RFC 9700 / OAuth 2.1): use PKCE for both.

  

    
On the wire — front channel vs back channel

    
Front channel = rides the browser's address bar (redirects — visible, interceptable). Back channel = a direct HTTPS call the user never sees. IdP endpoints live under /realms/KT-idp/protocol/openid-connect.

    
    
      
  • Front channel — GET /auth: the login redirect — this time with no code_challenge; returns with the one-time ?code=.
    • 
      
  • Back channel — POST /token from our server (/api/authcode-exchange): code + client_secret. The secret never enters the browser.
    • 
      
  • Back channel — POST /token from our server (/api/authcode-refresh): grant_type=refresh_token + the same secret.
    • 
      
  • Front channel — GET /logout: redirect that ends the IdP session.
