4 · Implicit (deprecated ☠)

OAuth flows  ·  public client kt-implicit

Tokens don't belong in URLs.

Why we show it: the old way SPAs worked before PKCE. The access token is returned directly in the URL fragment — so it leaks into history, referrer headers, and logs. And there's no code to protect. OAuth 2.1 removed it.

Uses response_type=token (not code) — that single change is the whole anti-pattern.

⚠ The access token arrived IN THE URL — look at your address bar

    
This is now in your browser history. That's exactly why Implicit is dead — the fix is flow #1, Authorization Code + PKCE.

On the wire — front channel vs back channel

    
Front channel = rides the browser's address bar (redirects — visible, interceptable). Back channel = a direct HTTPS call. IdP endpoints live under /realms/KT-idp/protocol/openid-connect.

    
    
      
  • Front channel — GET /auth (response_type=token): the login redirect.

  • Front channel — the return redirect carries the access token itself in the URL fragment (#access_token=…).
    • 
      
  • No back channel at all — and that's the flaw: tokens belong on the back channel. This flow ships them through the address bar, so they leak into history, logs and referrers.
